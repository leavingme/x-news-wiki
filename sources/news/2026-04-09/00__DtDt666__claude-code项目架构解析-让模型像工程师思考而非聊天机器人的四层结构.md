# Claude Code项目架构解析：让模型像工程师思考而非聊天机器人的四层结构

## 摘要

**[仓库结构决定模型行为上限：CLAUDE.md是记忆锚点，.claude/skills/是能力复用层]**

核心方法论：杂乱仓库→普通聊天机器人；结构清晰的仓库→扎根代码库的专属工程师。四个关键组件：1）CLAUDE.md=仓库核心记忆，强制精简（内容过长模型反而忽略关键信息）；2）.claude/skills/=可复用专业模式；3）结构地图=文件模块分布；4）规范准则=允许与禁止的操作边界。

本质是"给AI一份工程师能看懂的接手续任文档"。

**gakki：这套框架的深层逻辑是——模型能力的瓶颈从来不是模型本身，而是输入信息的结构化程度。大多数人的CLAUDE.md写成了用户手册而不是工程师的交接文档。**

## 原文

大多龙虾和Claude Code 玩家都觉得，用好龙虾的关键是写出更优质的提示词。

其实根本不是。
 
真正的核心，是把你的代码仓库梳理出清晰结构，让 Claude 能像工程师一样思考。
 
如果兄弟们的仓库杂乱无章，Claude 就只是个普通聊天机器人；
如果仓库结构规整，Claude 就会变成扎根在你代码库里的专属开发者。
 
一个项目只需要做好这四件事：
 
- 核心逻辑：系统的功能定位
- 结构地图：文件模块的分布位置
- 规范准则：允许与禁止的操作边界
- 工作流程：任务的执行方式
 
我把这套体系称作：
Claude Code 项目架构解析👇
 
1️⃣ CLAUDE.md = 仓库核心记忆（内容务必精简）
 
这个文件是 Claude 的行动总纲。
别写成长篇大论，只需要包含三点：
 
- 项目用途：系统存在的核心价值
- 仓库结构：项目的目录布局
- 规范+指令：Claude 的操作准则
 
一旦 CLAUDE.md 内容过长，模型反而会忽略关键信息。

清晰简洁，永远比篇幅冗长更重要。

2️⃣ .claude/skills/ = 可复用的专业模式
 
别再反复在提示词里重复指令了。
把常用工作流封装成可复用的技能模块即可。
举几个例子：
 
- 代码审查清单
- 代码重构手册
- 调试工作流程
- 版本发布规范
 
这样 Claude 就能一键切换到专业模式，
最终效果是：
跨会话、跨团队协作时，输出结果更稳定一致。

3️⃣ .claude/hooks/ = 行为护栏

大模型会遗忘规则，但钩子不会。
用钩子实现必须自动执行的固定操作。
示例：
 
- 代码编辑后自动运行格式化工具
- 核心模块修改后自动触发测试
- 屏蔽敏感目录（认证、计费、数据迁移相关）
 
钩子能让 AI 工作流，变成可靠的工程化体系。

4️⃣ docs/ = 渐进式上下文
 
别把所有信息都塞进提示词里，
让 Claude 自主查阅你的文档就够了。
示例：
 
- 架构总览
- 工程决策记录（ADR）
- 运维操作手册
 
Claude 不需要把所有内容记在脑子里，
它只需要知道权威信息的位置即可。

5️⃣ 核心模块的本地 CLAUDE.md
 
系统中有些模块暗藏复杂逻辑，
在这些关键位置添加本地上下文文件。
示例：
src/auth/CLAUDE.md
src/persistence/CLAUDE.md
infra/CLAUDE.md
 
这样 Claude 在处理这些高风险模块时，能精准理解其复杂性，
大幅降低出错概率。

大多数人都忽略了一个关键转变：
提示词是临时的，
结构才是长久的。
 
一旦你的代码仓库为 AI 量身设计，
Claude 就不再是普通聊天机器人……
而是化身融入项目的专属工程师。🚀
