# Goal 的真正价值在于「过程比答案更重要」的任务——写好 Goal 的核心是写交付物而不是写意图

## 摘要

**一句话核心判断：Goal 写得好不好，直接决定 Codex 是在持续工作还是在原地打转。**

Goal 最适合「过程比答案更重要」的任务——工具调研、多文件代码修改、附来源的教程写作。好的 Goal 写法要同时限定目标、信息源优先级、输出结构和不确定性处理方式。

反例：「帮我调研一下 Codex Goal」，这等于没写，Codex 不知道交付物是什么。

正例：「输出事实结论、用户教程、可发布正文草稿，所有关键判断附来源链接，无法确认处标注不确定」——这才是一个可执行的目标。

gakki锐评：Goal 模式本质上是把「产品需求文档」的能力要求转移给了用户。不会写 Goal 的人，大概率也写不好产品需求——这个门槛比想象中要高。

## 原文

最适合 Goal 的任务

Goal 最适合那些“过程比答案更重要”的任务。

比如你要做一个工具调研，不要只说：
帮我调研一下 Codex Goal。

更好的写法是：
/goal 请调研 Codex Goal mode 的最新变化，优先使用 OpenAI 官方文档、GitHub release 和 PR 作为来源。请输出三部分：事实结论、用户使用教程、适合发布到中文 AI 账号的正文草稿。所有关键判断都要附来源链接，无法确认的地方标注不确定。

这个写法的好处是，它同时限定了目标、来源、输出结构和不确定性处理方式。

如果是代码任务，可以这样写：
/goal 请阅读当前项目，找出登录页表单校验失败的原因，修复问题，补充必要测试，并在最后总结修改文件、验证命令和剩余风险。

这比“帮我修 bug”稳定很多，因为 Codex 知道最终交付物不是一段解释，而是修复、测试和总结。
