# Peter Steinberger 访谈：OpenClaw 与个人助理元年

## 摘要

**OpenClaw 创始人分享项目爆红背后的思考：从 CLI 到 MCP 的哲学之争与闭环原则。**
- **核心观点**：代码本身已不值钱，系统性思维才是核心竞争力；CLI 因其 Unix 哲学的可扩展性优于 MCP。
- **助理定义**：今年是个人助理元年，真正的 Agent 应该是运行在“你的机器、你的规则”下的虚拟同事。

**Pi 的锐评**：Peter 指出了大厂的死穴——数据主权。OpenClaw 的爆红，本质上是人们在 AGI 时代对个人隐私与自主权的最后夺取。

## 原文

最近爆紅的ClawBot aka Moltbot aka OpenClaw 的開源專案創作者 Peter Steinberger受訪時表示，AI 時代軟體開發正在發生根本性轉變，講了蠻多有趣的議題，包含他現在和agent協作的工作模式。

OpenClaw 這個開源個人 AI 助理專案在短短數週內爆紅，GitHub 星星數突破十萬，單週吸引兩百萬訪客。創辦人 Peter Steinberger 在多次訪談中分享了他對 AI 開發的獨特見解，以及軟體工程正在經歷的根本性轉變。

#從燃燒殆盡到重新發現熱情

Peter 曾經營 PSPDFKit 這家 PDF 框架公司 13 年，產品被超過十億台裝置使用，團隊成長到 70 人。但長期高強度工作讓他燃燒殆盡，賣掉公司後休息了三年。

2024 年四月，他重返科技圈。「當時 AI 還不錯，雖然不是很好。我想說為什麼沒人在討論這個？我覺得是因為我錯過了那三年 AI 很糟的時期。我回來的時候剛好趕上 Claude Code 在二月推出 beta 版，這是我的第一次體驗，然後我就睡不著了。」

他甚至創辦了一個聚會，最初叫做「Claude Code Anonymous」，現在改名為「Agents Anonymous」。「就像賭場的經濟學一樣。這是我的小型吃角子老虎機。你按下按鈕，叮叮叮叮叮，然後要麼是垃圾，要麼是真正讓你驚豔的東西。」

#OpenClaw的誕生：從 WhatsApp Relay 到開源平台

去年十一月，Peter 想要一個能隨時隨地與 AI agent 互動的方式。「我花了大約一小時，拼湊出一個 WhatsApp 整合，能接收訊息、呼叫 Claude Code，然後回傳結果。」

在馬拉喀什的一次旅行中，他無意間發送了一則語音訊息給這個簡陋的系統。「我根本沒有建構語音訊息的支援功能。但大約五到十秒後，我的 bot 像什麼事都沒發生一樣回覆了。」

Agent 解釋說它查看了檔案標頭，發現是 Opus 格式，用 ffmpeg 轉檔，找到環境裡的 OpenAI key，用 curl 發送到 OpenAI 取得翻譯。Peter 說這就是他頓悟的時刻：「這些東西真的是非常聰明、足智多謀的野獸，只要你真正給它們權力。」

專案經歷了多次更名：WhatsApp Relay → Claudius → ClawBot → MoltBot → OpenClaw。最終定名 OpenClaw：「Open 代表開源、對所有人開放、社群驅動；Claw 則是我們的龍蝦傳承。」

#2025年是個人AI助理元年

Peter 認為 AI 應用正在經歷重大轉折。「去年是 coding agent 變得真正好的一年。今年是個人助理的一年。未來，每個人都會有一個最好的朋友，而那是一台機器，它會理解你，會知道你的一切，可以為你做任務，會主動行動。」

他指出大型科技公司都在開發類似產品，但 OpenClaw 提供了一個關鍵差異：「你的助理、你的機器、你的規則。不像 SaaS 助理把你的資料放在別人的伺服器上，OpenClaw 在你選擇的地方運行。你的基礎設施、你的金鑰、你的資料。」

「想像一下，你不只是買一台新電腦，你有這個幽靈實體，你給它一個鍵盤和滑鼠，還有網路存取，它就像一個虛擬同事。任何你能在電腦上做的事情，那個 agent 都能做。」

#CLI優於MCP：Agent 工具調用的哲學之爭

Peter 對目前流行的 MCP（Model Context Protocol）持批評態度。「作為一個過渡工具還可以，我認為 MCP 最好的成果是讓公司們重新思考開放更多 API。但整個概念有點蠢。」

「你必須在 session 載入時預先匯出所有工具的所有函數和所有說明，然後模型必須發送精確的 JSON blob，然後得到 JSON 回傳。但你知道什麼真正能擴展嗎？CLI。Agent 懂 Unix，你電腦上可以有一千個小程式。」

他舉例說明 MCP 的限制：「想像你有一個天氣服務。模型要求可用城市的列表，得到 500 個城市回來，然後它必須從中選一個。但它無法過濾那個列表，因為這不是 MCP 的工作方式。而如果是 CLI，它可以用 jq 精確過濾它需要的東西。」

「我建了 Make Porter，一個小小的 TypeScript 工具，可以把 MCP 轉換成 CLI。所以現在你可以在手機上告訴 agent『用 Vercel MCP 做這件事』，它會去網站上找到 MCP，載入它，然後使用它，全部按需進行。」

#不同 AI 模型的實戰比較

Peter 分享了他對各種模型的評價：

#AnthropicOpus：「以相當大的領先優勢成為最好的。我不知道他們用什麼訓練，但它在 Discord 裡表現得太好了。它的行為很像人類，不會回覆每一則訊息。它真的會讓我笑出來，而 AI 的笑話通常都很糟糕。個性方面，Opus 是我的最愛。」

#OpenAICodex：「非常可靠，是更可靠的工作者。對於寫程式，我更喜歡 Codex，因為它能導航大型程式碼庫。你真的可以下 prompt 然後直接推送到 main branch，我有 95% 的把握它確實能運作。它會沉默地讀檔案 10 分鐘，而 Claude 會讀三個檔案就自信滿滿地開始寫程式碼。」

#GoogleGemini：「不是一個好模型，在 agentic 的意義上表現不好。我確定有方法可以為每個模型優化一點，但總體來說我沒有發現它在 tool calling 方面是個好模型。」

#MiniMax M2.1：「目前最具 agentic 特性的開源模型。如果你預算有限，MiniMax 是很好的替代方案，10 美元月費的方案大約等於 Anthropic 100 美元月費的請求量。」

#閉環原則：有效 AI 開發的核心秘密

Peter 強調「閉環」是讓 AI coding 真正有效的關鍵。「要有效使用 coding agent，你必須閉環。Agent 需要能夠自己除錯和測試。這就是大秘密。」

「這就是為什麼這些模型在寫程式碼方面這麼好，但在創意寫作方面可能只是一般般。因為沒有簡單的方法來驗證寫作，但程式碼我可以編譯、lint、執行、驗證輸出。如果你用正確的方式設計，你就有一個完美的迴圈。」

他舉了一個實際例子：「昨天我在 debug 一個 Mac app 找不到遠端 gateway 的問題，但同樣的 TypeScript 程式碼可以運作。Mac app 很煩人因為你要 build、啟動、查看、說這不對。

所以我告訴 agent：『你去建一個純粹用來 debug 的 CLI，調用所有相同的 code path，你可以自己呼叫，然後你就自己迭代修復。』然後它就自己煮了一小時，完成後告訴我：『這裡有 race condition，這裡有 misconfiguration。』聽起來合理，我不需要看那些程式碼。」

「使用 agentic coding 其實會讓你成為更好的程式設計師，因為你必須更努力思考你的架構，讓它更容易被驗證。」

#PullRequest已死，Prompt Request 萬歲

Peter 提出了一個大膽的觀點：「Pull request，我現在更把它們看作是 prompt request。」

「有人開一個 pull request，我不會逐行看程式碼。我說聲謝謝，然後思考這個功能，然後和我的 agent 一起從這個 PR 開始，按照我認為合適的方式設計這個功能。agent 很少重用程式碼，但它讓 agent 對目標有很好的理解。」

「在我的公司，有時候一個 pull request 要花一週的工作，你評論它，然後有人必須切換上下文，你等 CI 跑 40 分鐘。現在我有討論，我看看這會怎麼影響某些東西，我讓模型審查，我們把它重塑成符合我願景的形式，然後我們把程式碼織入。」

「我現在用這麼多新詞來描述用這些模型寫程式碼。比如把程式碼『織入』現有結構。有時候你必須改變結構讓它能容納新功能。」

關於 CI，他更直接：「我不太在乎 CI 了。我有本地 CI。Agent 會跑測試。如果測試在本地通過，我們就合併。Agent 把它叫做 gate，我不知道這名字從哪來的。『我要跑 full gate 嗎？』Full gate 就是 linting、building、checking、跑所有測試。我現在也開始用他們的語言了。」

#高度並行的新工作流程

Peter 描述了他的日常工作方式：「我沒有一個員工要管理，我有五到十個同時在做事情的 agent。我設計一個新的子系統或功能，然後我知道 Codex 大概需要 40 分鐘或一小時來建構。所以我要把計劃做對，然後建構它，然後轉移到別的事情上。」

「這個在cook，然後我做那個，然後那個在cook，然後這個在cook，然後我回到這個。我不斷在腦中跳來跳去。我希望我不需要這樣做，這大概是一個過渡期的問題，某個時候我們會有夠快的模型和系統，我可以少並行一點。但要保持在 flow state，我需要大量並行。」

這種方式讓他達到了驚人的產出。「我現在每天發布的程式碼比我公司一個月發布的還多。而我們當時有 70 個人。」

「有一天我有 600 個 commits。有人做了 code review 說『這實際上不是 slop』。是的，裡面有很多技術含量。這是很多努力的工作，但你需要玩這項技術，學習它，然後你會進步。」

#程式碼不值錢了，系統思維才重要

Peter 對程式碼價值的看法發生了根本轉變。「在我的公司，我曾經對每個細節都很執著，每個空格、每個換行、命名。我花了那麼多時間在 bikeshedding 上。回想起來，我想：搞什麼？客戶又看不到內部。」

「大多數 app 是什麼？資料從 API 進來是一種形式，你 parse 它，包裝成另一種形式，存到資料庫是另一種形式，出來又是另一種形式，然後是 HTML 或什麼的。你只是在整個 app 中把資料按摩成不同形式。我們基本上是美化版的印表機。而真正困難的部分 30 年前被 Postgres 的 nerds 解決了。」

「我不需要在乎這個按鈕怎麼對齊，或用哪個 Tailwind class。只要它看起來好就行。但我在乎 Telegram 的 pairing code 怎麼運作，這樣其他人不能冒充我驗證？是的，我會看那段程式碼。」

「你需要有系統理解。你可以不看某些東西，但你必須看其他東西。即使全部都是 AI 寫的，還是很多工作，因為這些 agent 還缺少的是願景、品味和愛。」

#大公司採用AI的困境

Peter 認為大公司會很難有效採用 AI。「這也需要完全重新定義公司的運作方式。在 Google 他們告訴你，你要麼是工程師，要麼是經理。但這個新世界需要的人是有產品願景、能夠做所有事情的人，而且需要更少的這種人，但他們要有非常高的主動性和能力。」

「如果我要重建 PSPDFKit，我可以輕鬆用 30% 的人來經營公司。這從經濟上來說很可怕，很多人會在這個新世界中難以找到位置。但我一點都不驚訝現有公司無法非常成功地使用 AI。」

「你必須先做一個大重構，不只是 codebase，還有你的公司。我設計 codebase 不是為了對我有用，而是要對 agent 容易。我為不同的東西優化，不總是我偏好的東西，而是我知道對這些模型最有效、摩擦最少的東西。因為最終是它們要處理程式碼，不是我。」

#為什麼資深開發者在抗拒 AI

Peter 觀察到兩種截然不同的開發者類型：

「有些人更關心結果、產品。我很在乎它的感覺和一切，但底層的管道怎麼運作，我在結構上在乎，但不到最細節的程度。」

「然後有些人真的喜歡在困難問題上寫程式碼，想演算法，不太喜歡『我在建一個產品，要做所有行銷』的部分，他們更喜歡解決困難問題。這些人真的很掙扎，經常拒絕 AI 或變得很沮喪，因為那正是 AI 做的工作：解決困難問題。」

他批評了一位知名開發者的測試方法：「他寫了一個 prompt，放到 Claude web，按發送，然後拿輸出來跑，它沒編譯過，他就失望了。但當然它第一次不會成功。你認為我能在第一次嘗試就寫出沒 bug 的程式碼嗎？」

「這些模型是我們人類集體知識的幽靈。它們在很多方面工作方式類似。當然你第一次不會做對，這就是為什麼你要閉合回饋迴圈。而且你不只是發一個 prompt 給模型，你開始一段對話。」

#PromptInjection與安全挑戰

隨著專案爆紅，安全性成為首要關注。「我為自己建了這個，用於一對一的 WhatsApp 或 Telegram。現在人們把它用在不受信任的環境中，所有我原本不在乎的威脅模型現在都出現了。」

「老實說，這大部分是 vibe coded 的。裡面有不少 agentic engineering，但最終我想建造的是展示給人們看的東西，不是企業公司的完成品。Prompt injection 還沒有被解決，確實存在風險。」

Peter 用他的 soul 檔案作為 pen testing：「在 OpenClaw 裡有一個系統，agent 有 identity 檔案、memory 檔案，還有一個 soul 檔案，是你放入 agent 應該如何思考和互動的價值觀的地方。我保持它關閉，部分是因為這是我 0.00001% 的秘密，部分是因為這是一個 pen testing 測試。到目前為止沒有人 exfiltrate 過 Claude 的 soul，很多人試過了，這給了我一些信心。」

「如果我用一個非常小或舊的模型，你問它足夠多次，最終它會很樂意給你一切。但用最新一代的模型，我有信心你必須非常非常努力才能 exfiltrate 它。」

#對新手的建議

對於想進入這個領域的人，Peter 建議：「我會建議他們要有無限的好奇心。是的，進入這個市場會更難，你需要建造東西來獲得經驗。我不認為你需要寫很多程式碼，但你需要去看看那些複雜的開源專案，學習它們。」

「你有一台無限耐心的機器，能夠解釋所有事情給你聽，你可以問所有問題：為什麼要這樣建構？來獲得系統理解。但這需要真正的好奇心。」

「他們也有優勢，就是他們沒有被所有經驗汙染。他們用我們甚至想不到的方式使用 agent，因為他們不知道那不會成功，而到那時候它可能已經成功了。」

#開源的未來：社群驅動的 AI 助理

Peter 強調 OpenClaw 的開源本質：「我全心希望這個專案能比我活得更久。我需要幫助，一個人做太多工作了。我只能這麼久不睡覺。」

「我希望有維護者加入。如果你熱愛開源，如果你有經驗，如果你喜歡處理 security report，或者喜歡拆解軟體然後幫忙改進而不只是丟工作給我，請來 Discord。」

關於開源授權，他說：「這會發生，有人會拿去賣。但最終這真的不重要。程式碼已經不值那麼多錢了，你可以刪掉然後幾個月內重建。真正有價值的是想法、關注度，也許還有品牌。」

「龍蝦已經蛻變成最終形態了。歡迎來到 OpenClaw。對了，吉祥物還是龍蝦。有些東西是神聖的。🦞」
