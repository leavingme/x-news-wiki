# Codex多线程管理：TUI正在被自己革命

## 摘要

**【一句话核心判断】**
Codex刚上线的"一个聊天管理其他聊天线程"功能，本质上是让AI操控自己的界面，标志着TUI范式正在被GUI化的多线程交互所替代。

**要点：**
- 新功能：一个Codex聊天可创建、检查、pin、归档、组织子线程，还能为并行任务拉起独立worktree
- 使用场景：扫描GitHub bug报告→在不同worktree中创建线程修复→重要线程置顶，随时归档review
- 核心逻辑：消除频繁切换窗口导致的心智状态丢失，用户只需盯住一个聊天窗口

gakki锐评：TUI的"克制"正在被自己吃掉。Codex不是在改良TUI，而是在用AI重新定义界面交互方式——让Agent操作界面，而不是让用户操作Agent。这个转变比功能本身更重要。

## 原文

以Claude code为代表的TUI模式正在Codex打败

Codex刚刚出了一个功能，能让你管理多个聊天的对话的效率翻倍。

他们让Codex的一个聊天支持，管理其他聊天线程。管理线程可以创建、检查、pin、归档、组织，使用子线程。

这意味着你只要盯住一个聊天窗口工作就行，需要的时候再切换到对应的聊天窗口。避免你在频繁切换窗口和记忆失去心理状态。

如何使用呢？在任意先窗口输入：

扫描 GitHub 上的 bug 报告，然后针对看起来比较容易修复的 bug，在不同的 worktree 中创建新的线程去修复它们。对于最重要的任务，也把这些线程置顶。

你就能看到Codex开始创建新线程分配worktree，然后将重要线程置顶。这个和那种用完就放弃的Subagent完全不同，你可以将这些线程归档，随时review。

随着Codex的崛起，TUI这种违背用户体验的模式将会慢慢被抛弃。

因为人的心智资源是有限的，任何消耗心智资源的设计当有新的方案的时候，都会被慢慢替代。

Codex真的越来越好用了！！
