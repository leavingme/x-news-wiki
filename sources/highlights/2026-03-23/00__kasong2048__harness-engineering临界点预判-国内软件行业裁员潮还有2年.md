# Harness Engineering临界点预判：国内软件行业裁员潮还有2年

## 摘要

**[范式迁移进入倒计时]** 卡颂提出AI Coding三次范式变迁论：从AI辅助编程→Vibe Coding→Harness Engineering。当前国外已进入第三阶段——程序员不写代码，只设计可观测、可评估、可回退的编码环境。

• 国内主流仍停留在第二阶段，大规模应用滞后国外约2年
• Harness Engineering核心逻辑：AI写出bug时，优先修复环境而非指导AI
• 可观测性成关键瓶颈，Agent Browser等基础设施是前置条件

赛博朋克2077正在提前到来。当编码环境的约束足够强，程序员的角色将从"写代码"彻底转向"设计游戏规则"。问题不是会不会裁员，而是被裁的那批人是否意识到自己已经站在悬崖边。

## 原文

距离国内软件行业大裁员还有接近 2 年的时间 

为什么是接近 2 年呢？因为在AI领域，一个范式从 在国外出现，到圈内热议，再到国内公司大规模应用的周期，就是 2 年。

在国外 AI Coding 领域，这 3 年经历了3次范式变迁：
1. AI 辅助编程：将 AI 作为 Linter、搜索引擎、代码补全工具
2. Vibe Coding：与 AI 结对编程，也是大部分人用 Cursor 的方式
3.（当下）Harness Engineering：程序员设计强约束、可观测、可评估、可反馈、可回退的编码环境，由 Coding Agent 自主完成所有编程工作

3 的核心思想是 —— 程序员不写一行代码，如果发现 AI 写出不好的代码，第一要务不是指导他改，而是排查环境中哪一环出了问题，修复环境。

举个例子：当 AI 写的前端代码有 bug，首先应该排查是不是在”可观测性“上不足（比如接入 Agent Browser 让他能自己走查页面）

当前国内主流编程范式是 2，还是大量需要程序员参与。当 1 年多 2 年后，3 成为主流范式，就会出现程序员裁员潮

缺粮 10% 的结果不是粮食涨价 10% ，而是一直涨价到饿死 10% 人为止。同样，主流范式进入 3 会一直让“主要工作职责能被 3 能力范围（也就是业务开发的所有编码工作）覆盖的程序员”全部失业
